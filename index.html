What is this?
Project Overview
Watch Party lets users create virtual rooms, invite friends via a room code, and watch YouTube videos in perfect real-time sync. Play, pause, seek, or change a video — everyone in the room follows instantly.
🏠
Room-based model
Create or join rooms via unique 4-8 char codes like CAFE-4821. Rooms auto-expire after 24 hours.
⚡
Real-time sync
Socket.io events broadcast play/pause/seek to all clients in under 50ms. Latency compensation built-in.
🔐
Secure & role-gated
JWT auth + OTP email verification. Playback controls restricted to HOST and MODERATOR roles only.

System Design
Architecture Overview
How the frontend, backend, database, and real-time layer connect together.
Full System Architecture

Key Design Decision: No raw video streaming

Watch Party doesn't stream video bytes. Every client loads the same YouTube video independently via the IFrame API. The server only syncs playback commands (play/pause/seek/change). This means zero video bandwidth on your servers, infinite scalability for video delivery.

Security
Authentication Flow
Email OTP verification on signup, JWT for all protected routes and WebSocket connections.
Signup → OTP → JWT Flow
🔑 HTTP Auth
Every protected REST route passes through auth.middleware.ts. It extracts the Bearer token from the Authorization header, verifies with jwt.verify(), and attaches req.user with the decoded payload.
🔌 Socket Auth
socketAuth.middleware.ts runs on every new socket connection. Reads the token from socket.handshake.auth.token and verifies it before allowing the connection to proceed.
Core Feature
Room Lifecycle
From room creation to expiry — every state a room passes through.
Room State Machine
1

Create Room

User hits POST /room/create. Backend generates a unique uppercase room code (e.g. CAFE-4821), creates a Room document in MongoDB with status: "WAITING", and initializes it in the in-memory roomCache. Creator is assigned HOST role.

2

Join Room

Another user emits join_room with the room code. Server adds them to the roomCache, updates the DB, and emits room_data (participant list) to everyone. It also emits request_sync specifically to the HOST so the new joiner gets the current video timestamp.

3

Watch & Sync

HOST or MODERATOR controls video. Each action emits video_action → server validates permission → broadcasts receive_video_action to all clients in the room. A periodic sync_time heartbeat keeps latecomers in sync.

4

Disconnect Grace Period

On socket disconnect, the server waits 5 seconds before cleaning up. This prevents "ghost leaves" from React StrictMode double-mounts or brief network blips. If the socket reconnects within 5s, the user stays in the room seamlessly.

5

Room Expiry

Every room has an expiresAt field with a MongoDB TTL index set to 24 hours. MongoDB auto-deletes expired rooms without any cleanup cron job needed.

Core Algorithm
Video Synchronization
How every client stays in sync despite different network latencies.
Play Event: Host → Server → All Clients (with latency compensation)

Why latency compensation matters

If Host is in Mumbai and Client is in New York (say 180ms RTT), without compensation the New Yorker would be 90ms behind. With compensation: client calculates latency, adds it to the received timestamp, and seeks slightly ahead. Everyone ends up at the same frame.

Infinite loop prevention
When a client receives receive_video_action and calls player.seekTo(), the YouTube player fires its own onStateChange event — which would re-emit video_action back to server, creating a loop. YoutubePlayer.tsx uses a isRemoteAction ref flag to skip emitting when the seek was triggered by a socket event.
In-memory cache strategy
roomCache.ts keeps current room state in memory (a plain JS Map). Every seek/heartbeat hits cache only — not MongoDB. DB writes happen only on critical events: user join/leave, video change, and status change. This prevents thousands of DB writes per minute during active sync.
Access Control
Roles & Permissions
Four-tier RBAC system enforced on both frontend (UI disabled) and backend (socket events rejected).
Role Hierarchy & Permissions

Double-layer enforcement

Frontend disables buttons for restricted roles (PARTICIPANT can't see play controls). But this alone isn't enough — backend videoEvents.ts and roleEvents.ts also check the user's role from roomCache before processing any socket event, and silently reject unauthorized actions.

WebSocket API
Socket Event Reference
All Socket.io events, their direction, and payloads.
CLIENT → SERVER (emit these from frontend)
create_room C→S Creates room, initializes roomCache, assigns HOST role to creator
join_room C→S { roomId } — Adds user to room, triggers request_sync to HOST for new joiner
leave_room C→S { roomId } — Removes user after 5s grace period
video_action C→S { action: 'play'|'pause'|'seek'|'change', currentTime, videoId? } — HOST/MOD only
sync_time C→S { currentTime } — HOST heartbeat, broadcast to all with serverTimestamp
transfer_host C→S { targetUserId } — HOST only. Transfers HOST role to another participant
change_role C→S { targetUserId, newRole } — HOST only. Promotes/demotes users
kick_user C→S { targetUserId } — HOST only. Removes user and terminates their socket
send_message C→S { message } — Rate-limited. Broadcasts to all in room
SERVER → CLIENT (listen for these on frontend)
room_data S→C Updated participant list (socketId stripped for security) — emitted on join/leave/role change
receive_video_action S→C { action, currentTime, serverTimestamp } — Triggers local player seek + play/pause
request_sync S→C Sent specifically to HOST when a new user joins — host responds with current video state
you_were_kicked S→C Sent to kicked user before socket is terminated. Frontend shows message and redirects.
receive_message S→C { sender, message, timestamp } — Chat message broadcast to room
Code Deep Dive
Function Reference
What every major function does, where it lives, and what it emits.
🔌 sockets/socketHandler.ts — main entry point
initSocketHandler(io) socketHandler.ts

Called once when the server starts. Registers the socketAuth middleware, then calls roomEvents, videoEvents, and roleEvents handlers on every connection. Maintains the roomCache — a Map<roomId, RoomState> shared across all modules.

🏠 sockets/roomEvents.ts — join/leave/create
create_room handler roomEvents.ts

Creates a new Room document in MongoDB with status: "WAITING", generates a unique room code, adds it to roomCache, joins the socket to a Socket.io room (channel), assigns the creator as HOST.

emits room_created to creator with the room code.

join_room handler roomEvents.ts

Validates room exists and isn't closed. Adds user to roomCache[roomId].users array, updates MongoDB. Joins the socket to the Socket.io room channel.

emits room_data to all in room (updated participant list, socketId stripped).

emits request_sync to HOST socket specifically — so new joiner gets current video position.

disconnect + leave_room handler roomEvents.ts

On disconnect, sets a 5-second setTimeout before actually removing the user. If they reconnect within 5s (common with React StrictMode), the timeout is cleared. After 5s, user is removed from cache and DB.

emits room_data to remaining participants.

🎬 sockets/videoEvents.ts — playback control
video_action handler videoEvents.ts

Handles play, pause, seek, and change_video. First checks if the emitting socket belongs to a HOST or MODERATOR (from roomCache). If not, silently ignores. If yes, updates videoState in roomCache, stamps serverTimestamp: Date.now(), and broadcasts to room.

requires HOST or MODERATOR role.

emits receive_video_action to all sockets in room including self.

db write Only on change_video — updates videoState.youtubeId in MongoDB.

sync_time handler videoEvents.ts

Called periodically by the HOST as a heartbeat. Server reads currentTime from payload, adds serverTimestamp, and broadcasts to all clients in room. Clients use this to resync if they drifted (e.g. network hiccup).

emits receive_sync to all in room.

👑 sockets/roleEvents.ts — permissions management
change_role handler roleEvents.ts

HOST sends { targetUserId, newRole }. Server verifies requester is HOST. Updates role in roomCache and MongoDB. Broadcasts updated participant list.

requires HOST role.

emits room_data to all — so all clients update their UI (enable/disable controls).

kick_user handler roleEvents.ts

HOST sends { targetUserId }. Server finds the target's socketId from roomCache, emits you_were_kicked to that specific socket, then calls socket.disconnect(true) to terminate their connection.

requires HOST role.

emits you_were_kicked to target socket then terminates connection.

transfer_host handler roleEvents.ts

Current HOST passes HOST role to another participant. Current HOST is demoted to PARTICIPANT. New HOST gets full control.

emits room_data with updated roles to all.

⚛️ Frontend — Key Components
YoutubePlayer.tsx components/

Wraps react-youtube. Uses an isRemoteAction ref (boolean) to distinguish between user-initiated events and socket-triggered events. When a receive_video_action comes in, sets isRemoteAction = true, calls the player API, then sets it back to false — preventing the player's own onStateChange from re-emitting the event back to the server.

Dashboard.tsx pages/

The main hub after login. Manages the layout switch between "lobby view" (room list, create/join) and "inner room view" (video player + chat). Handles socket connection lifecycle — connects on mount, stores socket in context/state, disconnects on unmount.

useDebounce.ts hooks/

Custom hook that debounces a value by a given delay. Used in the video URL/ID input to prevent firing change_video socket events on every keystroke. Only emits after the user stops typing for the specified delay (e.g. 500ms).

Database Schemas
MongoDB collections and their structure.
registers collection
username:   String  // unique
email:      String  // unique, for OTP
password:   String  // bcrypt hashed
isVerified: Boolean // OTP completed?
createdAt:  Date
otpverifies collection
email:     String
otp:       String  // 6-digit code
expiresAt: Date    // TTL ~10 min
rooms collection — core schema
roomId:     String  // e.g. "CAFE-4821", unique
hostId:     ObjectId  // ref: Register
status:     'WAITING' | 'ACTIVE' | 'CLOSED'
users:      [{ userId, role, socketId, username, joinedAt }]
videoState: {
  youtubeId: String,    // YouTube video ID
  isPlaying: Boolean,
  currentTime: Number,  // seconds
  lastSyncAt: Date
}
settings:   { maxParticipants, allowChat, isPrivate }
expiresAt:  Date  // TTL index = 24 hours → auto-deleted

Why two separate caches?

roomCache (in-memory) is used for every seek/heartbeat operation (can be hundreds per minute). MongoDB (disk) is used for persistence — join/leave events, video changes, and status updates. This hybrid approach avoids overwhelming the database while ensuring room state survives server restarts (minus the last few unreported seeks).

WebSocket on Render — important!

Render supports WebSockets on all plan tiers including free. No special config needed. However, free-tier Render services spin down after inactivity. The first connection after spin-down may take 30–60 seconds. For production demos, use a paid tier or keep the service warm with a ping cron.

Getting Started
Local Setup
Run the full stack locally in under 5 minutes.
1

Clone the repository

git clone https://github.com/learner-dot-new/watch-party.git
cd watch-party
2

Setup Backend

cd backend
npm install

# Create backend/.env
PORT=3000
MONGODB_URI=your_mongodb_uri
JWT_SECRET=any_random_secret_string
EMAIL_USER=youremail@gmail.com
EMAIL_PASS=your_gmail_app_password
FRONTEND_URL=http://localhost:5173

npm run dev  # starts on :3000
3

Setup Frontend

cd ../frontend
npm install

# Create frontend/.env
VITE_API_URL=http://localhost:3000

npm run dev  # starts on :5173
4

Open and test

Visit http://localhost:5173. Sign up with an email (OTP will arrive), create a room, open a second browser tab/incognito window, join the same room — test sync by pressing play.

Gmail App Password (for OTP email)

Go to Google Account → Security → 2-Step Verification → App passwords. Generate a password for "Mail". Use that as EMAIL_PASS — your regular Gmail password won't work with Nodemailer.
